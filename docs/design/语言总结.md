# C^ 语言总结

## 1. 语言定位

C^ 是一门面向原生编译的强静态语言。它的目标不是简单复刻 C 或 C++，而是在保留底层控制能力的同时，提供更清晰的语义表达、更现代的控制流，以及更容易维护的工程体验。

从目前的语法和实现方向看，C^ 更适合被描述为：

> 一门强调副作用可见、类型语义清晰、兼顾底层能力与现代工程抽象的原生语言。

它更接近“现代系统语言 / 现代 C++ 替代探索”，而不是脚本语言或教学语言。

## 2. 语法风格概览

C^ 的整体语法风格可以概括为以下几点：

- 使用 `func` 作为函数声明入口
- 返回类型后置：`func add(int a, int b) -> int`
- 类型写法强调结构区分：数组、切片、指针、引用、右值引用、函数类型、元组、可空类型都各有明确语法
- 控制流倾向现代化：`match`、`defer`、`try/catch/throw`、`foreach`
- 泛型支持约束表达：`where`、`concept`、`requires`
- 支持类、结构体、接口、属性、模块与 FFI

## 3. 核心语法摘要

### 3.1 变量与绑定

```cpp
var x = 10;
let name = "c_hat";
const int MaxSize = 1024;
late File file;
```

- `var`：可变绑定
- `let`：不可重绑定
- `const`：编译期常量
- `late`：延迟初始化，先声明后构造

### 3.2 函数

```cpp
func add(int a, int b) -> int {
    return a + b;
}

func is_even(int x) -> bool => x % 2 == 0;
```

- 使用 `func` 统一函数声明风格
- 使用 `-> T` 指定返回类型
- 支持表达式函数体 `=>`
- 支持泛型、重载、默认参数、变参、`noexcept`

### 3.3 类型系统

```cpp
int value;
int[4] fixed;
int[] slice;
int^ ptr;
int& ref;
int!& readonly_ref;
string~ moved;
func(int, int) -> int op;
(int, string) pair;
int? maybe;
```

C^ 当前体现出的类型层次包括：

- 基础类型：整数、浮点、布尔、字符、空类型等
- 固定长数组与切片
- 指针与引用
- 右值引用 / 移动语义
- 函数类型
- 元组类型
- 可空类型
- 类、结构体、接口、枚举
- 泛型类型与约束

### 3.4 参数传递与副作用表达

```cpp
func inc(int& x) {
    x += 1;
}

func print_value(int!& x) {
    print(x);
}

func take(string~ s) {
    // 接管资源
}

inc(&value);
print_value(value);
take(text~);
```

这里体现了 C^ 一个非常重要的设计取向：

- 可变引用调用必须显式写 `&arg`
- 只读引用可以直接传值式调用
- 资源转移显式写 `arg~`

这样做的目标是让副作用和资源转移在调用点尽量可见。

### 3.5 控制流与错误处理

```cpp
if (x > 0) {
    print("positive");
}

match (token) {
    case IntLiteral(v) => handle(v),
    case Identifier(name) => handle_name(name),
    default => fallback()
}

try {
    risky();
} catch (Error e) {
    recover(e);
}

defer cleanup();
```

C^ 当前强调的控制流能力包括：

- `if / else`
- `for / while / do`
- `foreach`
- `match`
- `break / continue / return`
- `try / catch / throw`
- `defer`

### 3.6 面向对象与模块化

```cpp
interface Reader {
    func read(self!) -> string;
}

class FileReader : Reader {
    func read(self!) -> string {
        return "data";
    }
}

module net.http;
import net.io as io;
```

当前语言设计支持：

- `class / struct / interface / enum`
- 访问控制：`public / private / protected / internal`
- `virtual / override / abstract / final`
- 属性（getter / setter）
- 模块与导入
- `extern` 与 FFI

## 4. 核心设计原则

### 4.1 副作用应在调用点可见

这是 C^ 很有辨识度的一条原则。

- 修改外部对象时，调用点必须显式写 `&`
- 转移资源时，调用点必须显式写 `~`
- 只读访问保持简洁调用

目标是让代码评审和阅读时，能快速看出谁会被修改、谁会被消费。

### 4.2 默认安全，但保留底层能力

C^ 并不拒绝底层能力，而是希望先给出更安全、更清晰的表达层：

- 用 `match` 代替传统 `switch`
- 用 `defer` 管理作用域退出清理
- 用分层类型表达数组、切片、指针、引用
- 同时保留 `new/delete`、指针、`extern`、FFI

这是一种“先把语义讲清楚，再允许下沉到底层”的设计思路。

### 4.3 类型系统服务于表达清晰

C^ 明显倾向于把类型语义直接写在语法上。

例如：

- `int[4]` 与 `int[]` 的区别是固定数组与切片
- `int&`、`int!&`、`int~` 分别表达不同语义
- `func(...) -> T` 统一函数声明与函数类型的书写方式

目标是减少依赖隐式规则和上下文猜测。

### 4.4 现代控制流优先于历史包袱

C^ 不把兼容传统 C 语法作为唯一目标，而是更倾向于：

- 用 `match` 做模式匹配
- 用 `defer` 管理清理逻辑
- 用 `late` 表达延迟初始化
- 用 `where / concept / requires` 做泛型约束

目标是提升语言的可推理性和可维护性。

### 4.5 工程可维护性优先于特性堆叠

C^ 不应该只是“特性越来越多”，而应该形成统一语言风格。

因此语言演进应尽量遵守：

- 同一种语义尽量只有一种主要写法
- 符号职责应尽量单一
- parser 支持的能力要尽快补齐 semantic 规则
- 文档要明确推荐用法，而不是只罗列语法可能性

## 5. 当前最有特色的设计点

在目前的设计中，比较有辨识度的点包括：

1. **副作用可见的引用调用规则**
2. **数组 / 切片 / 指针 / 引用的分层建模**
3. **`defer`、nullable、`Result/?` 这类工程特性组合**
4. **`func(...) -> T` 统一函数声明和函数类型**
5. **在保留底层能力的同时引入现代控制流与约束泛型**

## 6. 总结

C^ 已经不是一个只停留在“能写点语法糖”的实验，而是在尝试建立自己的语言性格。

它的方向很明确：

- 保留原生编译语言的底层能力
- 提高类型与副作用表达的清晰度
- 引入更现代的控制流与工程抽象
- 尝试形成比传统 C++ 更一致、更可维护的开发体验

如果后续持续收敛语义、统一符号职责、补强语义分析与规范文档，C^ 会比单纯“加功能”的语言项目更有长期价值。
