# 函数签名设计文档

## 1. 概述

在支持函数重载的编程语言中，编译器需要为每个函数重载生成唯一的内部名称，以便在代码生成阶段正确识别和调用。本文档讨论C-hat语言中函数签名的设计方案，比较不同方案的优缺点，并提出推荐方案。

## 2. 现有方案：下划线分割

### 2.1 实现方式

目前C-hat编译器使用下划线(`_`)作为函数名和参数类型的分隔符，例如：

- `add(int, int)` → `add_int_int`
- `add(double, double)` → `add_double_double`
- `add(int, double)` → `add_int_double`

### 2.2 优点

- 实现简单，易于理解
- 生成的函数名可读性较好
- 与C/C++的命名风格一致

### 2.3 缺点

- **命名冲突风险**：如果函数名本身包含下划线，可能与参数类型的分割混淆
  例如：函数`add_int(int)`会被生成为`add_int_int`，与`add(int, int)`的签名冲突
- **参数类型表示有限**：复杂类型的名称可能包含下划线，进一步加剧命名冲突

## 3. 候选方案

### 3.1 方案1：特殊字符分隔

使用特殊字符（如`@`和`#`）作为分隔符，例如：

- `add(int, int)` → `add@int#int`
- `add(double, double)` → `add@double#double`
- `add(int, double)` → `add@int#double`

#### 优点

- 避免了与函数名中下划线的冲突
- 分隔符明确，易于解析

#### 缺点

- 生成的函数名包含特殊字符，可能在某些平台或工具中不被支持
- 可读性较差

### 3.2 方案2：双下划线分割

使用双下划线(`__`)作为分隔符，例如：

- `add(int, int)` → `add__int__int`
- `add(double, double)` → `add__double__double`

#### 优点

- 仍然使用下划线，保持与C/C++风格的一致性
- 减少了与函数名中单个下划线的冲突

#### 缺点

- 仍然存在冲突风险，只是概率较低
- 函数名可能变得过长

### 3.3 方案3：数字前缀

为每个函数重载分配唯一的数字ID，例如：

- `add(int, int)` → `add_1`
- `add(double, double)` → `add_2`
- `add(int, double)` → `add_3`

#### 优点

- 完全避免了命名冲突
- 生成的函数名简洁

#### 缺点

- 函数名失去了参数类型信息，不利于调试和分析
- 需要维护函数ID的映射表，增加了实现复杂度

### 3.4 方案4：类型编码

使用简短的类型编码代替完整类型名，例如：

- `int` → `i`
- `double` → `d`
- `byte` → `b`

生成的函数名：

- `add(int, int)` → `add_ii`
- `add(double, double)` → `add_dd`
- `add(int, double)` → `add_id`

#### 优点

- 生成的函数名简洁
- 保留了参数类型的信息
- 减少了命名冲突的可能性

#### 缺点

- 需要维护类型编码表
- 对于复杂类型，编码可能不够直观

## 4. 方案比较

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| 下划线分割 | 实现简单，可读性好 | 可能与函数名冲突 | 简单项目，函数名不含下划线 |
| 特殊字符分隔 | 避免冲突 | 特殊字符可能不被支持 | 对命名冲突敏感的项目 |
| 双下划线分割 | 减少冲突，风格一致 | 仍有冲突风险 | 函数名可能含单下划线的项目 |
| 数字前缀 | 完全避免冲突 | 失去类型信息 | 函数重载非常多的项目 |
| 类型编码 | 简洁，保留类型信息 | 需要维护编码表 | 对函数名长度敏感的项目 |

## 5. 推荐方案

综合考虑各方案的优缺点，推荐采用**双下划线分割**方案，理由如下：

1. **兼容性**：仍然使用下划线，与C/C++的命名风格保持一致，避免引入特殊字符可能带来的兼容性问题
2. **冲突减少**：双下划线作为分隔符，大大减少了与函数名中单个下划线的冲突可能性
3. **可读性**：生成的函数名仍然具有较好的可读性，便于调试和分析
4. **实现简单**：修改现有代码的成本较低，只需将单下划线分隔改为双下划线

### 5.1 实现建议

- 将函数名与参数类型之间的分隔符改为双下划线
- 不同参数类型之间仍然使用单下划线分隔
- 例如：`add(int, int)` → `add__int_int`

### 5.2 特殊情况处理

- **函数名本身包含双下划线**：如果函数名本身包含双下划线，可能仍然存在冲突风险。这种情况下，可以考虑在函数名前添加特殊前缀，或者使用其他方案。
- **复杂类型**：对于复杂类型（如结构体、类），建议使用类型的完整名称，但需要确保类型名中不包含双下划线，或者对类型名进行适当的转换。

## 6. 结论

函数签名设计是支持函数重载的编译器中的重要环节。选择合适的函数签名方案需要考虑兼容性、可读性、冲突避免等多个因素。双下划线分割方案在这些因素之间取得了较好的平衡，是推荐的选择。

未来随着C-hat语言的发展，可能需要根据实际使用情况对函数签名设计进行进一步的调整和优化。
