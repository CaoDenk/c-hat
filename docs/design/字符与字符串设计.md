# C^ 字符与字符串设计文档

为了回应"字符大小固定"的限制，并支持多编码环境，C^ 采用了**基于前缀的字面量系统**和**逻辑/存储分离**的类型设计。

## 1. 核心哲学：逻辑与存储分离

我们必须区分"字符的逻辑概念"和"字符的物理存储"。

*   **逻辑字符 (`char`)**：统一为 **32位 Unicode 码点 (Code Point)**。这是处理文本逻辑（如判断大小写、数字、Emoji）的最小完整单元，避免了处理变长编码（如 UTF-8/16）的复杂性。
*   **物理存储 (`string`)**：默认采用 **UTF-8** 变长编码，以节省空间并保持 ASCII 兼容性。

## 2. 字符类型体系 (Character Types)

### 2.1 为什么 `char` 默认是 32 位？

这是一个经过深思熟虑的决定，旨在解决 C#/Java 时代遗留的**代理对 (Surrogate Pairs)** 痛点。

*   **C#/Java 的历史包袱**：它们诞生时，Unicode 字符集尚未超过 65535 个，因此选择了 16 位 `char` (UTF-16)。
    *   **后果**：随着 Emoji 和生僻字的普及，`char c = '😊'` 在 C# 中是非法的（Emoji 需要 2 个 `char`）。这导致 `string.Length` 返回的不是字符数，而是代码单元数，极易引发 Bug。
*   **C^ 的现代选择**：作为一门新语言，C^ 选择了与 Rust/Go/Swift 一致的路线——**`char` 代表一个真实的 Unicode 标量值 (Scalar Value)**。
    *   **优势**：逻辑完备，算法简单，无需处理代理对。
    *   **代价**：单个 `char` 占用 4 字节（但这通常发生在寄存器或栈上，影响微乎其微）。

### 2.2 类型列表

为了满足不同场景（特别是与 Windows API 或底层协议交互）的需求，我们提供特定宽度的字符类型。

| 关键字       | 位宽   | 语义              | 用途                                       |
| :----------- | :----- | :---------------- | :----------------------------------------- |
| **`char`**   | 32-bit | Unicode 码点      | **默认类型**。用于文本处理算法、逻辑判断。 |
| **`byte`**   | 8-bit  | UTF-8 单元 / 字节 | 用于二进制数据、UTF-8 字符串底层存储。     |
| **`char16`** | 16-bit | UTF-16 单元       | 用于 Windows API 交互、Java/C# 互操作。    |
| **`char32`** | 32-bit | UTF-32 单元       | 等同于 `char`，但在强调物理存储时使用。    |

### 2.3 字符字面量与前缀

通过前缀区分不同宽度的字符字面量：

```cpp
char c1 = 'A';       // 默认: char (32-bit), 值: 65
char c2 = '中';      // 默认: char (32-bit), 值: 20013

byte b = u8'A';      // 前缀 u8: byte (8-bit)
char16 c3 = u16'中'; // 前缀 u16: char16 (16-bit)
char32 c4 = u32'😊'; // 前缀 u32: char32 (32-bit)
```

## 3. 字符串字面量与类型推导

字符串字面量本质上是**只读数组**。通过前缀控制其编码格式和元素类型。

### 3.1 类型体系：字面量、视图与切片

在 C^ 中，我们严格区分"字面量类型"、"通用视图"和"切片"。

#### 1. `literalview` (编译器推导类型)
*   **定义**：字符串字面量（如 `"hello"`）的**原生类型**，编译器内置类型，不属于标准库。
*   **结构**：`struct { byte!^ ptr; usize length; }`。
*   **语义**：**"我指向静态常量区 (.rodata)，我是永生的。"**
*   **用途**：主要用于重载决议。例如 `string` 的构造函数重载 `string(literalview)` 可以触发 Static Mode 优化，而 `string(string_view)` 则必须拷贝。
*   **推荐命名**：使用 `literalview`。这个名字清楚地表明它是一个指向 Literal 的 View，且暗示了静态生命周期。

#### 2. `string_view` (通用参数类型)
*   **定义**：它是 `byte![]` (不可变字节切片) 的**类型别名**。
*   **结构**：`struct { byte!^ ptr; usize length; }` (与 `literalview` 内存布局完全一致)。
*   **语义**：**"我指向某块内存，我是只读的，但我不保证生命周期。"**
*   **用途**：函数参数的标准类型。`func process(string_view s)` 可以接受字面量、`string` 对象、`byte[]` 数组等任何形式的字符串。
*   **转换**：`literalview` -> 隐式转换 -> `string_view` (安全，因为静态生命周期 > 任意生命周期)。

#### 3. `byte[]` vs `string_view`
*   **本质上**：它们在 C^ 中是**同一个东西**。`string_view` 只是 `byte![]` 的别名。
*   **误区澄清**：`byte[]` / `string_view` 只是简单的胖指针 (Fat Pointer)，它们**不负责**管理生命周期，也不维护"是否只读/是否静态"的状态位。状态位由 `std.string` (所有者) 维护。

```cpp
// 类型关系图
"hello" (Type: literalview) 
   |
   +--> 隐式转换 --> string_view (Type: byte![]) 
   |                  ^
   |                  | 隐式转换 (借用)
   +--> 构造 --> string (Owner)
```

### 3.2 栈上字符数组 (Stack Character Array)

为了明确区分**切片 (View)** 和**数组 (Copy)**，C^ 引入了 **`[$]`** 语法来显式表示"自动推导长度的栈数组"。

> **`[$]` 是全语言通用的语法糖**，不限于字符串。它的唯一语义是："编译器，请替我数一下初始化列表的元素个数，我不想自己数。" 效果等价于手写 `[N]`，零运行时差异。
>
> ```cpp
> int[$] arr = [1, 2, 3, 4, 5];   // 等价于 int[5]
> byte[$] buf = "hello";            // 等价于 byte[5]
> int[$, $] mat = [[1,2],[3,4]];   // 等价于 int[2, 2]（多维同样适用）
> ```

```cpp
// 1. 切片 (Slice) - 推荐
// s 指向 .rodata 中的常量，零拷贝
// 类型：byte![] (即 string_view)
byte![] s1 = "hello"; 
// s1[0] = 'H'; // Error: 指向只读内存

// 2. 栈数组 (Stack Array) - 显式拷贝
// [$] 是纯语法糖，编译器推导长度为 5，等价于 byte[5]
// 行为：栈上分配 5 字节，执行 memcpy("hello", s2, 5)
byte[$] s2 = "hello"; 
s2[0] = 'H'; // OK: 修改的是栈上的副本

// 3. 显式指定长度 (传统 C 风格)
byte[10] s3 = "hello"; // 剩余部分填 0
```

### 3.3 切片安全性与生命周期

C^ 的 `byte[]` 和 `string_view` 是非拥有视图。为了防止 Use-After-Free，编译器对**临时对象**到视图的转换有严格限制。

```cpp
func get_name() -> string { return "Alice"; }

// Error: 临时对象销毁，视图悬空
// byte[] view = get_name(); 

// Correct: 绑定到变量延长生命周期
var s = get_name();
byte[] view2 = s; // OK
```

## 4. 字符串内部实现与内存模型 (Implementation Model)

针对用户关于 "const string 优化" 和 "C++20 COW" 的疑问，C^ 采用了 **混合存储模型 (Hybrid Storage Model)**。

### 4.1 三种存储模式

`string` 对象内部通过 `capacity` 字段的高位标记支持三种模式：

1.  **SSO (Small String Optimization)**：
    *   **场景**：短字符串（<= 23 字节）。
    *   **行为**：数据直接存储在栈上对象内部。
    *   **代价**：极快的栈拷贝。

2.  **Heap (Normal)**：
    *   **场景**：动态生成的长字符串。
    *   **行为**：堆分配，标准的所有权模型。

3.  **Static Mode（零拷贝借用）- C^ 的独特优势**：
    *   **场景**：由字面量初始化的字符串（含 `const string` 和 `var string`）。
    *   **行为**：`ptr` 直接指向 `.rodata` 段，`STATIC_FLAG = 1`。
    *   **析构**：No-op，无需释放（`.rodata` 生命周期等于进程）。
    *   **写入**：触发一次性 Promote 到 Heap Mode，之后与普通堆字符串完全相同。

### 4.2 Static Promote 机制详解

> **命名说明**：C^ 的这套机制被称为 **Static Promote**（静态提升），而非传统的 COW（Copy-On-Write）。
> 两者的本质差异见本节末尾对比表。

C^ 的 `string` 实现了高效的**惰性提升 (Lazy Promote)** 策略：数据初始时借用静态只读区，仅在真正需要写入时才"提升"为独占堆内存。这保证了在不修改数据时永远零拷贝。

#### 1. 状态位设计 (State Flags)

`string` 利用 `capacity` 字段的最高位作为 **`STATIC_FLAG`**。

| 模式 | `STATIC_FLAG` | `ptr` 指向 | 析构行为 |
|------|:---:|------|------|
| **Static Mode**（借用只读区） | 1 | `.rodata` | No-op（无需释放） |
| **Heap Mode**（独占堆内存） | 0 | 堆上分配的内存 | `free(ptr)` |

> 注：最高位复用意味着 `capacity` 的有效范围是 `[0, 2^63 - 1]` 字节（约 8 EB），实践中不构成限制。

#### 2. Promote 动作（静态 → 堆提升）

所有**可变方法**（`operator[]=`、`append`、`clear` 等）在修改数据前，**必须**先调用内部的 `promote()` 函数。这是从 Static Mode 进入 Heap Mode 的唯一路径，且是**单向的、不可逆的**。

```cpp
// 标准库内部伪代码
private func promote(usize required_cap = 0) {
    // 快速路径：已经是 Heap Mode 且容量足够，直接返回
    // 代价：一次位运算 + 一次整数比较，分支预测器极易命中
    if (!is_static() && self.capacity >= required_cap) {
        return;
    }

    // 慢速路径：需要从静态区迁移到堆（只会发生一次）
    usize new_cap = calculate_growth(max(self.length, required_cap));
    byte^ new_ptr = heap_alloc(new_cap);

    if (self.length > 0) {
        memcpy(new_ptr, self.ptr, self.length);
    }

    // Static Mode 下 ptr 指向 .rodata，不需要 free；Heap Mode 下才 free 旧内存
    if (!is_static()) {
        heap_free(self.ptr);
    }

    self.ptr      = new_ptr;
    self.capacity = new_cap;  // 清除 STATIC_FLAG，正式进入 Heap Mode
}
```

#### 3. 读写分岔 (Read/Write Fork)

| 场景 | 初始状态 | 读取 | 写入 | 最终状态 |
|------|---------|------|------|---------|
| `const string s = "..."` | Static | 直接读，0 开销 | 编译期禁止 | 永远 Static |
| `var s = "..."` | Static | 直接读，0 开销 | 触发一次 promote | Heap（之后不再 promote） |
| `var s = compute()` | Heap | 直接读，0 开销 | 直接写 | Heap |

**典型流程图（`var s = "long literal string"`）**：

```
初始化:  s → .rodata  [Static Mode, 0 堆分配]
          ↓
读取:    s[0]         [直接读，0 开销]
          ↓
首次写:  s[0] = 'A'
          → promote() 检测到 Static Flag
          → malloc + memcpy  ← 只发生这一次
          → s 进入 Heap Mode
          → 执行写入
          ↓
后续写:  s[1] = 'B'
          → promote() 检测到已是 Heap Mode
          → 立即返回（一次位运算，可忽略）
          → 执行写入
```

#### 4. 拷贝语义

`string` 的拷贝是**值语义深拷贝**，不存在多对象共享堆内存的情况：

```cpp
// 情况 A：从字面量拷贝
string a = "hello world";   // a: Static Mode
string b = a;               // b 也进入 Static Mode（复制 ptr/len，不分配堆）
                            // .rodata 永远存活，无需引用计数

// 情况 B：从堆字符串拷贝
string a = compute();       // a: Heap Mode
string b = a;               // b: 深拷贝，独立堆内存，与 a 完全无关

// 情况 C：移动（零开销）
string b = a~;              // b 接管 a 的内存，a 被置空，无任何拷贝
```

> **规则**：Static Mode 下的"拷贝"是零开销的（仅复制两个字段），因为 `.rodata` 不需要生命周期管理；
> Heap Mode 下的拷贝是深拷贝，每个 `string` 对象独占自己的内存。**没有引用计数，没有共享堆内存。**

#### 5. 为什么不用经典 COW？

经典 COW（如 C++98 `std::string`）让多个对象共享同一块**堆内存**，写入时才复制。这个方案在 C++11 之后已被标准明确禁止，原因如下：

| 对比维度 | 经典 COW（引用计数） | C^ Static Promote |
|---------|:-----------:|:-----------:|
| 多对象共享堆内存 | ✅ 是 | ❌ 否（独占） |
| 多线程安全 | ⚠️ 需原子操作 | ✅ 天然安全 |
| 拷贝开销 | O(1)（仅加引用计数） | O(1)（Static）/ O(N)（Heap） |
| 写入开销 | O(N)（检查引用计数 + 可能拷贝） | O(1)（Heap 模式下检查极廉价） |
| 语义透明度 | ⚠️ 隐式共享，难以预测 | ✅ 显式独占，行为可预测 |
| 实现复杂度 | 高（原子计数 + 线程同步） | 低（一个 flag 位） |

C^ 的设计取舍是：**用"Heap 拷贝不再廉价"换取"无引用计数、无原子操作、语义清晰"**。  
在需要廉价转移所有权时，使用 `~` 移动语义（O(1)，零拷贝）而不是依赖隐式的 COW。

### 4.3 性能闭环：C^ vs C++20

| 场景 (长字符串)           | C++20 `std::string`       | C^ `std.string`           | 胜出              |
| :------------------------ | :------------------------ | :------------------------ | :---------------- |
| **`string s = "..."`**    | Heap Alloc + Memcpy O(N)  | **Static Zero-Copy O(1)** | **C^ (优势)**     |
| **`const s = "..."`**     | Heap Alloc + Memcpy O(N)  | **Static Zero-Copy O(1)** | **C^ (大优势)**   |
| **读取**                   | 指针间接寻址              | 指针间接寻址              | 平手              |
| **首次写入（字面量来源）** | 直接写 O(1)               | Promote + Write O(N)      | C++ (微弱优势)    |
| **后续写入**               | 直接写 O(1)               | 直接写 O(1)               | 平手              |
| **深拷贝**                 | Heap Alloc + Memcpy O(N)  | Heap Alloc + Memcpy O(N)  | 平手              |
| **转移所有权**             | `std::move` O(1)          | `s~` 移动语义 O(1)        | 平手              |

**结论**：C^ 在最常见的"只读字面量"场景（配置、日志标签、错误信息等）下实现了 O(1) 初始化，而 C++ 是 O(N)。
唯一的代价是：从字面量初始化的 `var string` **首次写入**时需要一次 Promote（malloc + memcpy），比 C++ 多一次堆分配。
这个代价在"先初始化后修改"的场景下完全可以接受；如果确定会立刻修改，直接用 `String.withCapacity(n)` 预分配即可跳过 Promote。

## 5. 增强字面量与插值

### 5.1 标准前缀

| 前缀      | 示例        | 类型            | 编码       | 说明                                                                 |
| :-------- | :---------- | :-------------- | :--------- | :------------------------------------------------------------------- |
| **(无)**  | `"你好"`    | `literalview`   | **UTF-8**  | **默认推荐**。编译器内置类型，指向 `.rodata`，永生，零拷贝。可隐式转换为 `byte![]`（`string_view`）。 |
| **`$`**   | `$"你好"`   | 由目标类型决议  | **UTF-8**  | **插值字符串**。通过 `operator "$"` 决议 Handler 类型，默认推导为 `std.string`。 |
| **`u16`** | `u16"你好"` | `char16[]`      | **UTF-16** | 对应 Windows `L"..."`。                                              |
| **`u32`** | `u32"你好"` | `char[]`        | **UTF-32** | 固定宽度。                                                           |
| **`b`**   | `b"\xFF"`   | `literalview`   | **Binary** | 原始字节，不校验编码，同样指向 `.rodata`。                           |

> **注意**：`literalview` 与 `byte![]`（`string_view`）的区别：
> - `literalview` 是**编译器内置类型**，携带"静态生命周期"语义，只能来自字符串字面量，任何类型都**不能**转换成它。
> - `byte![]` 是通用只读切片，可接受字面量、堆字符串借用、栈数组借用等任意来源。
> - `literalview` 可以**隐式向下转换**为 `byte![]`，反向禁止。

### 5.2 字符串插值 (`$`) 与 Handler 模式

#### 设计原则

C^ 的字符串插值基于 **Handler 模式**：编译器只负责将 `$"..."` 展开成一组固定的方法调用，
缓冲区的分配策略完全由 Handler 类型决定。

**关键设计**：编译器通过 `operator "$"` 的重载决议确定 Handler 类型，**不绑定任何具体类型名**。
标准库的 `string` 只是恰好重载了 `operator "$"`，它没有任何特权——任何类型都可以重载，
替换掉标准库也完全没有问题。

这与运算符重载（`operator "+"` 等）的机制完全一致，不引入任何新约定。

#### `operator "$"` 重载协议

任何希望支持 `$"..."` 语法的类型，需在其上定义一个静态工厂函数：

```cpp
// 协议：重载 operator "$"，返回一个 Handler 对象
public static func operator "$"(usize literal_len, usize hole_count) -> HandlerType;
```

编译器收到两个编译期常量：
- `literal_len`：所有字面量片段的字节长度之和
- `hole_count`：插值表达式的个数

Handler 对象需实现三个固定方法：

| 方法 | 签名 | 说明 |
|------|------|------|
| `append_literal` | `func append_literal(literalview s)` | 追加字面量片段 |
| `append_value` | `func append_value<T: Formattable>(T val, literalview fmt = "")` | 追加插值表达式 |
| `result` | `func result() -> R`（R 任意） | 返回最终结果，R 即插值表达式的推导类型 |

#### 编译器展开规则

```cpp
// 用户写：
string msg = $"User {name} is {age:D2} years old.";

// 目标类型是 string → 查找 string 上的 operator "$"
// 编译器展开为：
var __h = string.operator "$"(20, 2);   // literal_len=20, hole_count=2
__h.append_literal("User ");
__h.append_value(name);
__h.append_literal(" is ");
__h.append_value(age, "D2");            // 格式说明符作为 literalview 传入
__h.append_literal(" years old.");
string msg = __h.result();
```

> **格式说明符 `"D2"` 是 `literalview`**，编译期常量，零运行时解析开销。

**无目标类型时的推导规则**（和 `operator "+"` 重载决议一致）：

```cpp
var x = $"Hello {name}";   // 无目标类型
// 1. 在当前作用域查找所有 operator "$" 的实现
// 2. 唯一实现 → 推导为该类型（通常是标准库 string）
// 3. 多个实现 → 歧义，编译错误，要求显式写出目标类型
```

#### 标准库实现（无特权，可完整替换）

```cpp
// std/core/string.ch  —— 标准库的 string 重载了 operator "$"
// 这不是编译器魔法，是普通的运算符重载

extension string {
    public static func operator "$"(usize literal_len, usize hole_count) -> StringInterpolationHandler {
        return StringInterpolationHandler(literal_len, hole_count);
    }
}

// Handler 实现：预分配缓冲区，通常只有一次堆分配
public struct StringInterpolationHandler {
    private var string _buf;

    public StringInterpolationHandler(usize literal_len, usize hole_count) {
        // 预分配：字面量长度 + 每个插值洞启发式预估 16 字节
        _buf = String.withCapacity(literal_len + hole_count * 16);
    }

    public func append_literal(literalview s) {
        _buf.append(s);
    }

    // T 必须实现 Formattable，编译期类型检查，无装箱
    public func append_value<T: Formattable>(T val, literalview fmt = "") {
        val.format_into(&_buf, fmt);   // 直接写入缓冲，零中间分配
    }

    public func result() -> string {
        return _buf~;   // 移动返回，零拷贝
    }
}
```

#### `Formattable` 接口

```cpp
// 任何类型只要实现此接口，就能用于 $"..." 插值
public interface Formattable {
    // 直接追加到 buf，避免产生临时 string
    func format_into(string!& buf, literalview fmt);
}

extension int: Formattable {
    public func format_into(string!& buf, literalview fmt) {
        byte[20] tmp;                        // 栈缓冲，不触发堆分配
        usize len = int_to_str(self, tmp, fmt);
        buf.append(tmp[0..len]);
    }
}

extension float: Formattable { ... }
extension string: Formattable { ... }
```

#### 性能特性

| 场景 | 堆分配次数 | 说明 |
|------|:---:|------|
| 所有插值值都较短（≤16 字节） | **1 次** | 预估充足，无 realloc |
| 某个值超出预估 | **2 次** | `_buf` 触发一次 1.5x realloc |
| 无插值（`$"plain text"`） | **1 次** | 退化为普通 string 构造 |
| `const` 上下文（全编译期常量） | **0 次** | 编译器折叠为 literalview |

#### 自定义 Handler 示例

由于标准库没有任何特权，自定义 Handler 与标准库地位完全平等：

```cpp
// SQL 参数化查询（防注入）
public struct SqlHandler {
    private var string _sql;
    private var List<DbValue> _params;

    public SqlHandler(usize literal_len, usize hole_count) {
        _sql    = String.withCapacity(literal_len + hole_count * 2);
        _params = List.withCapacity(hole_count);
    }

    public func append_literal(literalview s) { _sql.append(s); }

    public func append_value<T: DbSerializable>(T val, literalview fmt = "") {
        _sql.append("?");           // 写占位符，不拼入实际值（防注入）
        _params.add(val.to_db());
    }

    public func result() -> PreparedQuery {
        return PreparedQuery(_sql~, _params~);
    }
}

// 让 PreparedQuery 支持 $"..." 语法
extension PreparedQuery {
    public static func operator "$"(usize literal_len, usize hole_count) -> SqlHandler {
        return SqlHandler(literal_len, hole_count);
    }
}

// 使用：和普通插值语法完全一致，目标类型决定用哪个 Handler
var id = 42;
PreparedQuery q = $"SELECT * FROM users WHERE id = {id} AND active = {true}";
// q.sql    → "SELECT * FROM users WHERE id = ? AND active = ?"
// q.params → [42, true]
```

```cpp
// 零堆分配日志（直接写入外部缓冲区）
public struct LogHandler {
    private var _writer: !& LogWriter;

    public LogHandler(usize literal_len, usize hole_count, !& LogWriter w) {
        _writer = w;
    }

    public func append_literal(literalview s) { _writer.write(s); }
    public func append_value<T: Formattable>(T val, literalview fmt = "") {
        val.format_into(_writer.buf(), fmt);
    }
    public func result() { _writer.flush(); }   // result() 返回 void 也合法
}
```

### 5.3 原始字符串与多行文本

*   **原始字符串 (`#`)**：`#"C:\Windows"#`，无需转义。
*   **多行字符串 (`"""`)**：自动处理缩进。
*   **组合使用**：`$#"""..."""` (插值 + 原始 + 多行)。

### 5.4 自定义前缀 (User-defined Prefix)

#### 为什么是前缀，而不是后缀？

C++ 的用户自定义字面量（UDL）使用**后缀**语法，如 `"{}"_json`。这个设计有一个根本缺陷：

> 编译器必须先把整个字符串 token 化，才能知道该用哪个 Handler。

这导致了三个连带问题：

1. **插值无法实现**：`"{name}"_json` 里的 `name` 不是变量，只是字符串内容。后缀在字符串解析完之后才介入，根本来不及决定展开策略，因此 C++ UDL 天然不支持插值。

2. **转义规则无法定制**：后缀 Handler 拿到的是已经按标准规则处理过的字符串，`"\d+\.\d+"_re` 里的 `\.` 需要写成 `"\\."`——因为标准转义发生在 Handler 介入之前。前缀 Handler 在词法层面就接管了字符串，可以自定义转义规则（如 `$re"\d+\.\d+"` 无需双重转义）。

3. **命名空间污染**：`_s`、`_sv`、`_ms` 等后缀在全局命名空间里争夺短名字，第三方库容易冲突。前缀与运算符重载体系统一，作用域规则一致。

C^ 选择**前缀**语法，编译器读到前缀就确定了 Handler 类型，字符串内容按该 Handler 的规则解析：

```
$xml"<root>{name}</root>"
 ↑
 词法阶段已知 Handler = XmlBuilder
 → 字符串内容可按 XML 规则处理
 → 插值孔 {name} 可正确展开
```

---

C^ 支持三类字符串前缀，它们互相独立、互不冲突：

---

**① 非插值前缀**：接收字面量原始字节，返回目标类型。适合正则、路径、字节序列等。

```cpp
// 定义
public static func operator "re"(literalview raw) -> Regex { ... }
public static func operator "path"(literalview raw) -> Path { ... }

// 使用
var r = re"\d+\.\d+";
var p = path"C:/Users/foo";
```

---

**② 通用插值前缀 `$`**：在目标类型上重载 `operator "$"`，通过目标类型推导决定 Handler。
适合"类型即语义"的场景——目标类型本身就能表达意图。

```cpp
// 标准库 string 重载了 operator "$"，所以是默认推导结果
var msg = $"Hello {name}";              // 推导为 string

// 让自定义类型也支持 $"..."
extension PreparedQuery {
    public static func operator "$"(usize literal_len, usize hole_count) -> SqlHandler {
        return SqlHandler(literal_len, hole_count);
    }
}
PreparedQuery q = $"SELECT * FROM users WHERE id = {id}";
```

**`operator "$"` 重载决议规则**：

| 场景 | 行为 |
|------|------|
| 有目标类型 | 调用目标类型的 `operator "$"` |
| 无目标类型，作用域内唯一实现 | 推导为该实现的 `result()` 返回类型 |
| 无目标类型，多个实现 | 编译错误：歧义，要求显式写出目标类型 |

---

**③ 具名插值前缀 `$prefix`**：重载 `operator "$prefix"`，前缀本身表达领域语义。
适合"前缀即领域"的场景——语义一眼可见，无需写出目标类型。

```cpp
// 定义：$xml 前缀
public static func operator "$xml"(usize literal_len, usize hole_count) -> XmlBuilder {
    return XmlBuilder(literal_len, hole_count);
}

// 定义：$sql 前缀
public static func operator "$sql"(usize literal_len, usize hole_count) -> SqlHandler {
    return SqlHandler(literal_len, hole_count);
}

// 使用：前缀即语义，无需目标类型标注
var node = $xml"<root><item>{name}</item></root>";  // 明确是 XML
var q    = $sql"SELECT * FROM users WHERE id={id}"; // 明确是 SQL
```

---

**三类前缀的优先级规则**：

```
$prefix"..."  →  直接调用 operator "$prefix"，不走目标类型推导
$"..."        →  走目标类型推导，调用目标类型的 operator "$"
prefix"..."   →  直接调用 operator "prefix"（非插值）
```

三者不会产生歧义：编译器先看 `$` 后面是否跟着标识符，有则走具名前缀路径，无则走通用插值路径。

> **标准库无特权**：无论是 `operator "$"` 还是 `operator "$prefix"`，都是普通的运算符重载。
> 标准库的 `string` 只是恰好重载了 `operator "$"`，编译器不关心类型叫什么名字。
> 替换标准库时，只需让你自己的字符串类型重载 `operator "$"` 即可。

## 6. 标准库字符串类型 (`std`)

### 6.1 `string` (UTF-8)
核心字符串类型。
*   **底层**：`byte[]` + `capacity` + `length`
*   **编码**：强制 UTF-8 有效性。
*   **构造**：`string s = "Hello";` (隐式转换 `literalview` -> `string`)

### 6.2 `u16string` 与 `wstring` (UTF-16)

#### 设计原则

C^ 只有**一个** UTF-16 字符串类型：`u16string`。`wstring` 是它在 Windows 平台上的别名，**不是独立类型**。

```cpp
// 标准库定义（伪代码）
public struct u16string { ... }   // 唯一的 UTF-16 字符串类型

// 仅 Windows 平台
#[platform(windows)]
public alias wstring = u16string;  // 零成本别名，编译后完全等同
```

#### 为什么有两个名字

| 名字 | 定位 | 适用场景 |
|------|------|---------|
| `u16string` | **正式类型名**，跨平台可用 | 通用 Unicode 互操作、跨平台代码 |
| `wstring` | Windows 平台别名 | 调用 Windows API（`CreateFileW`、`MessageBoxW` 等） |

两者**内存布局完全相同**，互转零开销——因为它们根本是同一个类型，只是名字不同。

```cpp
// Windows 平台上两者可以互换使用
u16string path = u"C:\\Users\\foo";
wstring   wpath = path;    // 零拷贝，直接赋值（同一类型）

// 传给 Windows API
CreateFileW(wpath.ptr, ...);   // wpath.ptr 是 char16^，即 LPCWSTR
```

#### 与 C 的 `wchar_t` 的关系

C 的 `wchar_t` 是平台相关的：Windows 是 16 位，Linux/macOS 是 32 位。C^ **不跟 `wchar_t` 绑定**：

- C^ 的 `char16` 永远是 16 位（对应 C 的 `char16_t`，不是 `wchar_t`）
- C^ 的 `u16string` 底层是 `char16[]`，永远是 UTF-16
- 与需要 `wchar_t*` 的 C 库互操作时，Windows 上直接传 `wstring.ptr`（16位相同），Linux 上需要手动转换

```cpp
// 底层结构
public struct u16string {
    private var char16[] _data;   // char16 = uint16，永远 16 位
    // ...
}

// 互转示例（零开销，同类型）
func takes_wstring(wstring s) { ... }
u16string s = u"Hello";
takes_wstring(s);    // OK，wstring 和 u16string 是同一类型
```

#### 字面量

```cpp
u16string s1 = u"Hello";         // u 前缀，编译器生成 UTF-16 编码
wstring   s2 = u"Windows路径";   // 同上，Windows 平台
```

## 7. 遍历与操作 (Iteration)

同一个字符串可以有不同的"视图"。

### 7.1 切片操作 (Slicing)

C^ 支持使用区间语法对字符串进行**零拷贝切片**。切片结果总是 `string_view` (即 `byte![]`)，绝不产生堆分配。

#### 倒数索引 `$expr`

在 `[...]` 下标表达式内部，`$expr` 表示**倒数索引**，等价于 `len - expr`：

```
s[$expr]  →  s[s.len - expr]
```

`expr` 可以是任意整数表达式（字面量、变量、运算式均可）：

```cpp
string s = "Hello World";   // len = 11

s[$1]       // s[11-1] = s[10]，最后一个字节 'd'
s[$2]       // s[11-2] = s[9]，倒数第2个
s[$i]       // s[11-i]，i 是变量
s[$i + 1]   // s[11-(i+1)]
```

**`$expr` 的作用域限制**：`$expr` **只在 `[...]` 内部合法**，在其他位置出现是编译错误。
这条规则保证了 `$` 的双重语义（插值前缀 vs 倒数索引）在词法/语法层面完全不相交：

| 语法形式 | 语义 | 合法位置 |
|---------|------|---------|
| `$"..."` 或 `$prefix"..."` | 字符串插值前缀 | 字符串字面量之前 |
| `$expr` | 倒数索引，等价于 `len - expr` | **只在 `[...]` 内** |

**边界情况**：

```cpp
s[$0]       // 任何位置都是编译错误：$0 无意义，"倒数第0个"不对应任何有效元素
s[6..$0]    // 同上，编译错误。"从6到末尾"请写 s[6..]（省略右端点）
```

> `$N` 要求 N ≥ 1。编译期能确定 N = 0 时直接报错；运行期表达式求值为 0 时触发边界检查（Debug panic，Release UB）。

#### 切片示例

```cpp
string s = "Hello World";   // len = 11

// 1. 基础切片（返回 string_view，零拷贝）
var sub1 = s[0..5];      // "Hello"（字节 0,1,2,3,4）
var sub2 = s[6..11];     // "World"

// 2. 倒数索引切片（$N = len - N，N 必须 ≥ 1）
var sub3 = s[1..$1];     // "ello Worl"（去掉首尾各1字节，即 s[1..10]）
var sub4 = s[$2..$1];    // "ld"（s[9..10]，倒数第2到倒数第1）

// 3. 开放区间（省略端点 = 取到末尾 / 从头开始）
var sub5 = s[6..];       // "World"（从索引6到末尾，等价于 s[6..len]）
var sub6 = s[..$1];      // "Hello Worl"（从头到倒数第1个，即 s[0..10]）
var sub7 = s[..];        // 整个字符串的视图（等价于 s[0..len]）

// 错误示例
// s[6..$0]  // 编译错误：$0 无意义，开放区间请用 s[6..]

// 4. 变量倒数索引
var n = 3;
var tail = s[$n..];      // 去掉前3个字节，"lo World"
var mid  = s[$n..$n];    // 中间段（去掉前后各n字节）
```

> **注意**：切片操作基于**字节**（O(1)），不检查 UTF-8 边界。
>
> #### UTF-8 字节切片越界行为
>
> 若切片索引落在 UTF-8 多字节字符的中间（例如汉字占 3 字节，但只切了前 2 字节），C^ **不插入任何校验**，这是**用户的责任（Undefined Behavior at language level）**：
>
> - 编译器和运行时均不检查 UTF-8 字节边界，零额外开销
> - 切出的 `byte![]` 是合法的字节切片，但其内容可能是不完整的 UTF-8 序列
> - 后续若将该切片解释为字符串（如打印、传给 `string` 构造函数），行为取决于接收方的容错能力
> - 这是**有意为之**：字节切片 `s[m..n]` 是纯粹的字节操作，和 `memcpy` 同等地位，C^ 不强制 Unicode 正确性
>
> **安全替代方案**：若需按 Unicode 码点安全切片，请先获取 `chars` 视图再切：
>
> ```c^
> var s = "Hello 你好";
> var sub = s.chars[6..8];  // 按字符切，保证 UTF-8 完整性（返回 "你好"）
> ```

### 7.2 视图遍历

#### `s.bytes` 与 `s.chars` 的本质

| 属性 | `s.bytes` | `s.chars` |
|------|-----------|-----------|
| 元素类型 | `byte`（原始字节） | `char`（Unicode 码点） |
| 随机访问 `[i]` | O(1) | O(N)，需从头扫描 |
| `s.chars` 本质 | — | **惰性迭代器**，不预分配，按需解码 |
| 长度 `len` | O(1)，字节数 | O(N)，需完整扫描 |

> **`s.chars` 是惰性的**：访问 `s.chars[i]` 时，解码器从字节 0 开始扫描，找到第 i 个 UTF-8 序列才停止。它不会 eager 解码全部字符，也不缓存结果。
>
> 如果需要频繁按字符索引访问，先收集到数组：`char[$] chars = s.chars.toArray();`，此后 `chars[i]` 是 O(1)。

```cpp
string s = "A中😊";
//          1字节 3字节 4字节，共8字节，3个字符

// 1. 字节视图 - O(1) 随机访问
s[0]          // byte 'A'
s.bytes[0]    // 同上
s.len         // 8（字节数）

// 2. 字符视图 - 惰性迭代器，顺序遍历 O(N)
for (char c : s.chars) { print(c); }  // A 中 😊

// 3. 字符随机访问 - O(N)，每次从头扫描
s.chars[0]    // 'A'（扫描0步）
s.chars[1]    // '中'（扫描1个字符）
s.chars[2]    // '😊'（扫描2个字符）

// 4. 字符数 - O(N)
s.chars.count()   // 3（扫描整个字符串）

// 5. 需要频繁按字符索引访问时，先物化为数组
char[$] arr = s.chars.toArray();   // 一次 O(N) 扫描
arr[2]   // O(1)，'😊'
```

#### 如何选择 `s[i]`（字节索引）vs `s.chars[i]`（字符索引）

| 场景 | 推荐 | 原因 |
|------|------|------|
| 协议解析、二进制处理、网络数据 | `s[i]` / `s.bytes` | 字节是实际单位，O(1) |
| 顺序处理 Unicode 文本 | `for c in s.chars` | 惰性，零额外开销 |
| 需要字符随机访问（编辑器光标等） | 先 `.toArray()` 再 `[i]` | 避免每次 O(N) 扫描 |
| 切片（截取子串） | `s[byte_start..byte_end]` | 始终基于字节，O(1) |

## 8. 高级模式匹配与编译期优化 (Advanced Features)

### 8.1 字符串模式匹配
C^ 支持在 `match` 语句中对字符串进行高效匹配，包括字面量匹配和守卫子句 (Guard Clauses)。

```cpp
match (filename) {
    // 1. 精确匹配 (O(1) via Hashing)
    case "README.md" => open_readme(),
    
    // 2. 多项匹配
    case "LICENSE" | "COPYING" => show_license(),
    
    // 3. 守卫子句 (灵活匹配)
    case s if s.starts_with("config_") => load_config(s),
    case s if s.ends_with(".json")     => parse_json(s),
    
    case _ => error("Unknown file type")
}
```

### 8.2 编译期哈希 (Compile-time Hashing)

为了保证性能，编译器对全部 case 均为字符串字面量的 `match` 块进行特殊优化：

1. **哈希计算**：编译器在编译期计算所有 case 字符串字面量的哈希值（内部使用 FNV-1a 或 CityHash，编译器自行选择，用户不可配置）。
2. **代码生成**：将字符串匹配转换为整数 `switch` 指令，运行时只做一次哈希 + 整数比较。
3. **冲突处理**：见下节。

这意味着在 C^ 中使用字符串 `match` 几乎与整数 `match` 一样快。

#### 哈希冲突处理

**冲突处理是编译器的正确性保证，不是实现细节。**

无论编译器内部用什么哈希算法、是否发生冲突，`match` 的语义永远正确——编译器负责保证这一点，用户不需要关心。

具体机制：

- **无冲突**（大多数情况）：整数 `switch` 直接命中，O(1)。
- **有冲突**（同一哈希值对应多个 case）：编译器在命中该哈希桶后，自动生成**二级字符串逐字节比较**，确认是哪个 case，正确性不受影响，性能略有下降但仍为 O(len)。

```
match (s) {
    case "foo" => A,   // hash("foo") = 0x1234
    case "bar" => B,   // hash("bar") = 0x5678
    case "baz" => C,   // hash("baz") = 0x5678  ← 与 "bar" 冲突

    // 编译器生成（伪代码）：
    // switch (hash(s)) {
    //   case 0x1234: if s == "foo" => A; else => _
    //   case 0x5678: if s == "bar" => B;        ← 冲突桶，生成二级比较
    //               else if s == "baz" => C;
    //               else => _
    // }
}
```

> **用户不能控制 hash 函数**。哈希函数是编译器内部实现细节，暴露给用户没有意义——冲突处理已经保证了正确性，而 hash 函数的选择对用户代码的可观察行为没有影响。
>
> 含有守卫子句的 case（如 `case s if s.starts_with(...)`）不参与哈希优化，始终顺序求值。

## 9. 实现揭秘：无黑魔法设计

C^ 编译器对字符串字面量的处理遵循以下协议：

1.  **语法分析**：`"..."` -> `StringLiteral` 节点。
2.  **类型检查**：`StringLiteral` -> `std.literalview` (标记类型)。
3.  **重载决议**：
    *   `string s = "..."` -> 匹配 `string(literalview)` -> 触发 Static Mode。
    *   `byte[] b = "..."` -> 匹配隐式转换 `literalview` -> `byte[]`。

这不需要新的关键字，利用重载决议实现了高效的字面量处理。
