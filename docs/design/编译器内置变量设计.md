# 编译器内置变量设计

## 1. 概述

编译器内置变量（Compiler Built-in Variables）是一组由编译器在编译期自动注入的特殊变量，用于提供编译期元信息，如源码位置、调试信息、编译配置等。这些变量不需要用户声明，可以在代码中直接使用。

## 2. 设计目标

1. **零依赖**：内置变量不依赖标准库，由编译器直接提供
2. **编译期求值**：所有内置变量都是编译期常量，可在 comptime 块中使用
3. **类型安全**：每个内置变量都有明确的类型
4. **命名空间隔离**：使用特殊前缀避免与用户变量冲突
5. **可扩展**：设计应允许未来添加更多内置变量

## 3. 内置变量命名方案

### 3.1 方案对比

| 方案 | 示例 | 优点 | 缺点 |
|------|------|------|------|
| `__xxx__` | `__line__`, `__file__` | C/C++ 传统，用户熟悉 | 容易与宏冲突，不够优雅 |
| `@xxx` | `@line`, `@file` | 简洁，类似 C# | 与属性语法冲突 |
| `$xxx` | `$line`, `$file` | 简洁，Shell 风格 | 不够直观 |
| `__xxx` | `__line`, `__file` | Rust 风格 | 仍需双下划线 |
| `builtin.xxx` | `builtin.line`, `builtin.file` | 命名空间清晰 | 需要额外语法支持 |

### 3.2 推荐方案：`__xxx` 形式

采用 Rust 风格的单下划线前缀：

```c^
// 源码位置
__line      // 当前行号 (int)
__column    // 当前列号 (int)
__file      // 当前文件名 (literalview)
__function  // 当前函数名 (literalview)
__module    // 当前模块名 (literalview)

// 编译信息
__timestamp // 编译时间戳 (long)
__build_mode // 编译模式：0=Debug, 1=Release (int)
__compiler_version // 编译器版本 (literalview)
```

## 4. 内置变量详细设计

### 4.1 源码位置变量

```c^
// 当前行号（从 1 开始）
int line = __line;

// 当前列号（从 1 开始）
int col = __column;

// 当前文件名（完整路径）
literalview file = __file;

// 当前函数名（如果不在函数中则为空字符串）
literalview func = __function;

// 当前模块名（如果不在模块中则为空字符串）
literalview mod = __module;
```

### 4.2 编译信息变量

```c^
// 编译时间戳（Unix 时间戳）
long ts = __timestamp;

// 编译模式
// 0 = Debug
// 1 = Release
int mode = __build_mode;

// 编译器版本字符串
literalview ver = __compiler_version;
```

### 4.3 调试辅助变量

```c^
// 完整的源码位置信息（格式：file:line:column）
literalview __location;  // "main.c^:10:5"

// 函数签名（包含返回类型和参数）
literalview __signature; // "func add(int a, int b) -> int"
```

## 5. 使用场景

### 5.1 日志和调试

```c^
func log(literalview message) {
    // 自动包含源码位置信息
    print("[{}:{}] {}", __file, __line, message);
}

func example() {
    log("Something happened");
    // 输出: [main.c^:10] Something happened
}
```

### 5.2 断言和错误处理

```c^
func assert(bool condition, literalview message = "") {
    if (!condition) {
        panic("Assertion failed at {}: {} - {}", __location, __function, message);
    }
}

func divide(int a, int b) -> int {
    assert(b != 0, "Division by zero");
    return a / b;
}
```

### 5.3 编译期条件

```c^
comptime {
    if (__build_mode == 0) {
        // Debug 模式下的额外检查
        // ...
    }
}
```

### 5.4 反射集成

```c^
attribute SourceLocation {
    literalview file;
    int line;
}

func traced() {
    // 编译期自动填充源码位置
    var loc = SourceLocation {
        file: __file,
        line: __line
    };
    // ...
}
```

## 6. 实现方案

### 6.1 词法分析

在 Lexer 中识别 `__` 前缀的标识符：

```cpp
// Lexer.cpp
if (current == '_' && peek() == '_') {
    // 可能是内置变量
    std::string name = scanBuiltinName();
    if (isBuiltinVariable(name)) {
        return Token(TokenType::BuiltinVar, name, line, column);
    }
}
```

### 6.2 语义分析

在 SemanticAnalyzer 中处理内置变量：

```cpp
std::shared_ptr<types::Type> 
SemanticAnalyzer::analyzeBuiltinVar(const std::string& name) {
    if (name == "__line" || name == "__column") {
        return types::TypeFactory::getPrimitiveType(types::PrimitiveType::Kind::Int);
    }
    if (name == "__file" || name == "__function" || name == "__module") {
        return types::TypeFactory::getPrimitiveType(types::PrimitiveType::Kind::LiteralView);
    }
    if (name == "__timestamp") {
        return types::TypeFactory::getPrimitiveType(types::PrimitiveType::Kind::Long);
    }
    // ...
}
```

### 6.3 代码生成

在 LLVMCodeGenerator 中生成常量值：

```cpp
llvm::Value* 
LLVMCodeGenerator::generateBuiltinVar(const std::string& name) {
    if (name == "__line") {
        return llvm::ConstantInt::get(llvm::Type::getInt32Ty(context()), 
                                       currentLocation.line);
    }
    if (name == "__file") {
        return builder()->CreateGlobalStringPtr(currentLocation.file);
    }
    // ...
}
```

## 7. 与其他语言的对比

| 语言 | 行号 | 文件名 | 函数名 |
|------|------|--------|--------|
| C/C++ | `__LINE__` | `__FILE__` | `__func__` |
| C# | 无内置 | 无内置 | `MethodBase.GetCurrentMethod()` |
| Rust | `line!()` | `file!()` | `function_name!()` |
| Python | `inspect.currentframe()` | 同左 | 同左 |
| Go | 无内置 | 无内置 | 无内置 |
| Zig | `@line()` | `@src().file` | `@src().fn_name` |
| **C-Hat** | `__line` | `__file` | `__function` |

## 8. 扩展设计：内置函数

除了变量，还可以提供内置函数：

```c^
// 编译期类型检查
bool __is_comptime(T value);      // 检查是否是编译期常量
bool __is_same_type(T a, U b);    // 类型相等检查
bool __is_assignable(T from, U to); // 可赋值检查

// 编译期类型信息
typeinfo __type_of(T value);      // 获取类型信息
literalview __type_name(T type);  // 获取类型名称

// 编译期字符串操作
literalview __concat(literalview a, literalview b); // 字符串拼接
```

## 9. 完整示例

```c^
module example;

attribute Trace {
    literalview file;
    int line;
    literalview function;
}

func logged_add(int a, int b) -> int {
    // 自动记录调用位置
    var loc = Trace {
        file: __file,
        line: __line,
        function: __function
    };
    
    print("[{}] {} called at {}:{}", 
          __timestamp, loc.function, loc.file, loc.line);
    
    return a + b;
}

func main() -> int {
    var result = logged_add(1, 2);
    
    // 编译期条件
    comptime {
        if (__build_mode == 0) {
            print("Debug mode: {}", __location);
        }
    }
    
    return 0;
}
```

## 10. 总结

编译器内置变量设计采用 `__xxx` 形式的命名，提供源码位置、编译信息等元数据。这些变量：

1. 由编译器在编译期自动注入
2. 都是编译期常量，可在 comptime 中使用
3. 不依赖标准库，零依赖
4. 类型安全，有明确的类型
5. 可扩展，方便未来添加更多内置变量

这种设计既保持了与 C/C++ 的相似性，又具有 Rust 的简洁性，同时完全符合 C-Hat 的零依赖设计理念。
