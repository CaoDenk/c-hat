# C^ 高级特性开发方案

本文档详细说明三个高级特性的实现方案：协程、反射、编译期执行。

---

## 一、协程开发方案

### 1.1 概述

C^ 协程采用无栈协程设计，基于契约驱动。核心关键字：`await`、`yield`。

### 1.2 实现阶段

#### 阶段一：基础 AST 支持（预计 2-3 天）

**目标**：添加协程相关 AST 节点和词法单元

**任务清单**：

1. **词法分析器修改**
   - 添加 `Await` 关键字到 [TokenType.h](file:///d:/projects/CppProjs/c-hat/src/lexer/TokenType.h)
   - `yield` 关键字已存在，确认可用

2. **AST 节点**
   - 已有 [YieldStmt.h](file:///d:/projects/CppProjs/c-hat/src/ast/statements/YieldStmt.h)，需确认完整性
   - 新增 `AwaitExpr` 表达式节点：
     ```
     src/ast/expressions/AwaitExpr.h
     src/ast/expressions/AwaitExpr.cpp
     ```
   - 新增 `CoroutineHandle` 内置类型支持

3. **NodeType 更新**
   - 在 [NodeType.h](file:///d:/projects/CppProjs/c-hat/src/ast/NodeType.h) 添加 `AwaitExpr`

#### 阶段二：解析器支持（预计 3-4 天）

**目标**：解析 `await` 和 `yield` 语法

**任务清单**：

1. **await 表达式解析**
   - `await` 是一元操作符，优先级与 `unary` 相同
   - 修改 [Parser.cpp](file:///d:/projects/CppProjs/c-hat/src/parser/Parser.cpp) 的 `parseUnaryExpr()`

2. **yield 语句解析**
   - `yield expr;` 或 `yield;`
   - 修改 `parseStatement()` 添加 yield 分支

3. **协程函数识别**
   - 检测函数体是否包含 `await` 或 `yield`
   - 标记函数为协程函数

#### 阶段三：语义分析（预计 4-5 天）

**目标**：验证协程契约和类型检查

**任务清单**：

1. **Promise 类型查找**
   - 在返回类型中查找内嵌 `Promise` 类型
   - 验证 Promise 契约方法：
     - `get_return_object()`
     - `initial_suspend()`
     - `final_suspend()`
     - `return_value()` / `return_void()`
     - `unhandled_exception()`
     - `yield_value()` (可选)

2. **Awaitable 契约验证**
   - 验证 `await` 操作数是否满足 Awaitable 契约：
     - `is_ready() -> bool`
     - `on_suspend(CoroutineHandle)`
     - `on_resume() -> T`

3. **协程状态机分析**
   - 识别挂起点
   - 分析变量生命周期跨越挂起点的情况

#### 阶段四：代码生成（预计 5-7 天）

**目标**：生成协程状态机 LLVM IR

**任务清单**：

1. **状态机结构生成**
   - 生成协程帧结构体
   - 包含 Promise、局部变量、恢复点索引

2. **状态机转换**
   - 将协程函数转换为状态机
   - 每个挂起点对应一个状态

3. **恢复函数生成**
   - 生成 `resume()` 函数
   - 生成 `destroy()` 函数

#### 阶段五：标准库支持（预计 2-3 天）

**目标**：提供基础协程类型

**任务清单**：

1. **Generator<T>**
   - 实现生成器 Promise
   - 实现迭代器接口

2. **Task<T>**
   - 实现异步任务 Promise
   - 实现同步等待

3. **内置 Awaitable**
   - `SuspendAlways`
   - `SuspendNever`

## 二、反射开发方案

### 2.1 概述

C^ 反射是纯静态的编译期机制，核心语法：`@Type`、`@typeof(expr)`、`comptime for`。

### 2.2 实现阶段

#### 阶段一：AST 支持（预计 2 天）

**目标**：完善反射表达式 AST

**任务清单**：

1. **ReflectionExpr 完善**
   - 已有 [ReflectionExpr.h](file:///d:/projects/CppProjs/c-hat/src/ast/expressions/ReflectionExpr.h)
   - 添加元对象属性访问支持

2. **新增 AST 节点**
   ```
   src/ast/expressions/FieldAccessExpr.h    // obj.[field]
   src/ast/expressions/TypeofExpr.h         // typeof(expr)
   ```

3. **ComptimeForStmt**
   - 扩展 [ComptimeStmt.h](file:///d:/projects/CppProjs/c-hat/src/ast/statements/ComptimeStmt.h)
   - 添加 `comptime for` 支持

#### 阶段二：解析器支持（预计 3 天）

**目标**：解析反射语法

**任务清单**：

1. **`@` 操作符解析**
   - `@TypeName` - 类型反射
   - `@typeof(expr)` - 表达式类型反射

2. **元对象属性访问**
   - `@T.name`、`@T.fields`、`@T.size` 等

3. **`comptime for` 解析**
   - 遍历 `@T.fields` 或 `@T.methods`

4. **`obj.[field]` 解析**
   - 动态字段访问语法

#### 阶段三：语义分析（预计 4-5 天）

**目标**：编译期元数据收集和验证

**任务清单**：

1. **类型元数据收集**
   - 在语义分析阶段收集类型信息
   - 存储字段、方法、属性等元数据

2. **元对象类型系统**
   - 定义编译期元对象类型
   - 支持属性访问类型检查

3. **`comptime for` 展开**
   - 验证循环变量类型
   - 准备编译期展开

4. **`comptime if` 支持**
   - 编译期条件分支

#### 阶段四：编译期执行引擎（预计 5-7 天）

**目标**：实现编译期代码执行

**任务清单**：

1. **常量求值器扩展**
   - 支持反射表达式求值
   - 支持 `comptime for` 展开

2. **元对象运行时表示**
   - 设计编译期元对象的数据结构
   - 支持字段/方法遍历

3. **类型比较**
   - 实现 `field.type == typeof(T)` 比较

#### 阶段五：Attribute 支持（预计 2-3 天）

**目标**：自定义编译期元数据

**任务清单**：

1. **Attribute 定义**
   - `comptime struct` 定义 Attribute

2. **Attribute 使用**
   - `[AttributeName]` 语法解析
   - Attribute 附加到类型/字段/方法

3. **Attribute 查询**
   - `field.has_attr(A)`
   - `field.attr(A)`

## 三、编译期执行开发方案

### 3.1 概述

`comptime` 关键字标记编译期执行的代码块，支持可变局部变量和复杂逻辑。

### 3.2 实现阶段

#### 阶段一：AST 支持（预计 1-2 天）

**目标**：完善编译期语句 AST

**任务清单**：

1. **ComptimeStmt 扩展**
   - 已有 [ComptimeStmt.h](file:///d:/projects/CppProjs/c-hat/src/ast/statements/ComptimeStmt.h)
   - 支持嵌套语句

2. **ComptimeExpr**
   - `comptime { ... }` 表达式形式
   - 返回编译期计算结果

#### 阶段二：解析器支持（预计 2 天）

**目标**：解析 comptime 语法

**任务清单**：

1. **comptime 块解析**
   - `comptime { statements }`
   - `comptime if (...) { ... }`
   - `comptime for (...) { ... }`

2. **comptime 表达式解析**
   - `const x = comptime { return value; }`

#### 阶段三：编译期执行引擎（预计 7-10 天）

**目标**：实现编译期代码执行器

**任务清单**：

1. **常量求值器架构**
   ```
   src/semantic/ConstEvaluator.h
   src/semantic/ConstEvaluator.cpp
   ```
   - 支持基本表达式求值
   - 支持控制流语句
   - 支持局部变量

2. **执行环境**
   - 编译期内存模型
   - 变量作用域管理
   - 函数调用支持

3. **限制与安全**
   - 执行步数限制
   - 内存限制
   - 禁止副作用操作

#### 阶段四：类型系统集成（预计 3-4 天）

**目标**：编译期类型作为一等公民

**任务清单**：

1. **类型值表示**
   - 类型作为编译期值
   - 类型比较操作

2. **泛型与 comptime 结合**
   - `comptime if` 用于泛型特化
   - 编译期类型分发

#### 阶段五：代码生成集成（预计 2-3 天）

**目标**：将编译期结果嵌入生成代码

**任务清单**：

1. **常量嵌入**
   - 编译期计算结果作为常量

2. **代码展开**
   - `comptime for` 展开为顺序代码
   - `comptime if` 移除死代码

## 四、依赖关系与实现顺序

### 4.1 特性依赖图

```
反射 (Reflection)
    │
    ├── comptime for ──┐
    │                  │
    └── typeof ────────┤
                       │
                       ▼
              编译期执行 (Comptime)
                       │
                       │
                       ▼
                 协程 (Coroutine)
              (需要 comptime if 做泛型特化)
```

### 4.2 推荐实现顺序

1. **第一阶段**：反射基础
   - AST 支持
   - 解析器
   - 基本语义分析

2. **第二阶段**：编译期执行基础
   - 常量求值器
   - comptime 块
   - comptime if

3. **第三阶段**：反射完善
   - comptime for
   - 元对象遍历
   - Attribute 支持

4. **第四阶段**：编译期执行完善
   - comptime for 展开
   - 类型系统集成
   - 代码生成集成

5. **第五阶段**：协程实现
   - AST 和解析器
   - 语义分析
   - 代码生成

## 五、风险评估

### 5.1 技术风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 协程状态机生成复杂 | 高 | 参考 Clang/LLVM 协程实现 |
| 编译期执行死循环 | 中 | 设置执行步数上限 |
| 反射元数据内存管理 | 中 | 使用编译期内存池 |
| 类型作为一等公民 | 高 | 设计清晰的类型值表示 |

### 5.2 时间估算

| 特性 | 预计时间 | 优先级 |
|------|----------|--------|
| 反射基础 | 10-12 天 | 高 |
| 编译期执行基础 | 12-15 天 | 高 |
| 反射完善 | 5-7 天 | 中 |
| 编译期执行完善 | 5-7 天 | 中 |
| 协程实现 | 15-20 天 | 中 |

**总计**：约 47-61 天
