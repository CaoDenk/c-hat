## 字符串插值 Handler 模式设计（2026-03-27 晚）

**决策**：`$"..."` 插值采用 Handler 模式（方案 C），不用两趟计算（方案 A）和增长式追加（方案 B）
**理由**：编译器不开洞，Handler 是普通标准库类型，与 LiteralView 设计哲学一致

**编译器展开规则**：
- 两个编译期常量：`literal_len`（字面量片段总长）、`hole_count`（插值洞数量）
- 展开为：构造 Handler → append_literal/append_value × N → result()
- 格式说明符 `{val:D2}` 中的 `"D2"` 作为 LiteralView 传入，编译期常量

**标准库**：`StringInterpolationHandler(literal_len, hole_count)` 预分配 `literal_len + hole_count*16` 字节，通常只有 1 次堆分配

**`Formattable` 接口**：`format_into(!& buf: string, LiteralView fmt)`，直接写入缓冲区，零中间分配

## 插值协议最终方案：operator "$"（2026-03-27 晚）

**问题**：之前方案硬编码 StringInterpolationHandler 类型名，语言依赖标准库。

**最终决策**：用 `operator "$"` 运算符重载协议，与 `operator "+"` 机制完全一致，编译器不绑定任何具体类型名。

- 目标类型上定义 `public static func operator "$"(usize literal_len, usize hole_count) -> HandlerType`
- 编译器通过目标类型推导决议用哪个 Handler
- 无目标类型时查找作用域内唯一实现（标准库 string 重载了它，所以默认推导为 string）
- 多个实现时编译错误，要求显式写目标类型

**三类前缀最终方案（§5.4）**：
1. `prefix"..."` → `operator "prefix"(LiteralView)` 非插值前缀
2. `$"..."` → 目标类型上的 `operator "$"(literal_len, hole_count)` 通用插值
3. `$prefix"..."` → `operator "$prefix"(literal_len, hole_count)` 具名插值前缀（$xml/$sql 等领域专用）
- 三类互不冲突，编译器看 `$` 后跟不跟标识符来区分②和③
- 标准库无特权，均为普通运算符重载
- **前缀设计动机**：C++ UDL 用后缀（`"..."_json`），必须先 token 化整个字符串才知道 Handler 类型，导致无法插值、无法定制转义规则、命名空间污染。C^ 前缀在词法阶段即确定 Handler，天然支持插值和自定义转义。

**文档改动**：`字符与字符串设计.md` §5.2 和 §5.4 全部重写

## 字符串设计补充（2026-03-27 晚）

**§7.2 s.chars 设计**：惰性迭代器，`s.chars[i]` 是 O(N)，不预分配不缓存。需频繁字符随机访问时先 `.toArray()` 物化为 `char[$]`，此后 O(1)。`s.chars.count()` 也是 O(N)。

**§8.2 match 哈希冲突**：
- 冲突处理是编译器正确性保证，不是实现细节
- 用户不可控制 hash 函数（编译器内部细节，不影响可观察行为）
- 冲突时生成二级字符串逐字节比较，正确性永远保证，性能略降但仍 O(len)
- 含守卫子句的 case 不参与哈希优化，顺序求值

**§6.2 wstring/u16string 设计**：
- `u16string` 是唯一的 UTF-16 类型，底层 `char16[]`（永远 16 位，不绑定 C 的 wchar_t）
- `wstring` 是 Windows 平台的 `alias wstring = u16string`，零成本别名，同一类型
- 互转零开销，Windows API 互操作直接用 `.ptr`

**文档改动**：`字符与字符串设计.md` §6.2、§7.2、§8.1/§8.2 全部扩写



**规则**：`$expr` 只在 `[...]` 下标表达式内合法，等价于 `len - expr`，expr 可以是任意整数表达式。

**作用域限制**：`$expr` 在 `[...]` 外部出现是编译错误，保证与插值前缀 `$"..."` 在词法上完全不相交。

**`$0` 处理（方案B）**：`$0` 在任何位置都是编译错误，不区分"单点"和"切片右端点"两种场景，规则统一无特例。开放区间用省略端点语法 `s[6..]` / `s[..$1]`。

## 术语统一决策（2026-03-27 晚）

**`LiteralView` → `literalview`（全小写无下划线）**
- 理由：内置类型一律全小写（与 `int`/`byte`/`char` 风格一致），用户定义类型用 PascalCase
- `StringView` 同时废弃，统一为 `literalview`
- 词法器早已使用 `"literalview"` 关键字，文档终于与实现保持一致
- **`lazy` 关键字不存在**，只有 `late`（延迟初始化）
- **宏支持**，但非必要场合不推荐使用
- C++ 编译器内部实现类名 `LiteralViewType` 保持 PascalCase（C++ 代码规范，不影响 C^ 语言层面）

**替换范围**：
- 文档：263 处 + 3 个文件名（docs/design/literalview设计.md，docs/dev/20260216-literalview系列）
- 源码：Lexer.cpp 调试字符串 `"LiteralView"` → `"literalview"`
- 命名规范.md：重写 §1.1，统一内置类型规则，删除"特殊内置类型 PascalCase"分类

**最终采纳方案 A**（与字符串字面量哲学一致）：

| 写法 | 类型 | 内存 | 备注 |
|---|---|---|---|
| `var arr = [1,2,3]` | `int![]` | `.rodata` | 默认只读切片，零运行时开销 |
| `int[$] arr = [1,2,3]` | `int[3]` | 栈 | 显式可变数组 |
| `int[3] arr = [1,2,3]` | `int[3]` | 栈 | 手动指定长度 |
| `var t = (1,2)` | `(int,int)` | 栈 | 元组始终是值类型 |

**关键决策**：`Type[$]` 语法**保留**，是全语言通用语法糖，让编译器替开发者数元素个数。

**重要设计决策（2026-03-27）**：

## 三个设计矛盾修复（2026-03-27 深夜）

### 1. UTF-8 字节切片越界 → UB（用户责任）
- `s[m..n]` 是纯字节操作，不检查 UTF-8 边界，零开销
- 切到多字节字符中间是**合法但用户责任（UB）**，行为取决于接收方如何处理字节序列
- 安全方案：`s.chars[i..j]` 按 Unicode 码点切（未来支持）
- 文档：`字符与字符串设计.md §7.1` 已补充详细说明

### 2. 数组是否有 `.ptr` 属性 → 有
- **数组和切片都有 `.ptr` 属性**（与 `.len` 对称）
- 数组的 `.ptr` 零额外开销（编译期已知首地址），等价于 `^arr[0]` 但语义更清晰
- 文档：`切片与数组属性及操作设计.md` §2.2、§3.2、§7.1 总结表、Q4 FAQ 全部修正

### 3. 反射 `@` 作用于类型还是值 → 永远作用于类型
- `@User` — 合法（User 是类型名）
- `@typeof(a)` — 合法（typeof先提取类型，@ 作用于类型）
- `@a` — **编译错误**（a 是值）
- 运行时反射通过 `obj.rtti.name` 等机制，不是 `@obj`
- 文档：`反射设计.md` 已正确，修正了 `spec/C^ 语言反射与元编程规范.md §4.3`

## 四个设计决策（2026-03-27 深夜续）

### 4. `len` 类型 → `long`（64位有符号）
- 有符号避免空数组 `len-1` 下溢（C++ `size_t` 是无符号，公认设计错误）
- `$expr` = `len - expr`，类型也是 `long`

### 5. `match` 范围 → 只支持左闭右开 `[m, n)`
- `90..101` 覆盖 90-100（满分 100 写 101）
- **不支持** `..=` 闭区间，语义单一
- 范围分支不允许重叠，编译器检查穷举
- 文档：`条件语句.md §2.2.3` 已更新

### 6. 后缀 `!` 强制解包设计
- 只作用于 `T?` 或 `Result<T,E>`，其他类型使用 `!` 是**编译错误**（类型前置，编译器在解析时已知类型）
- 运行时失败：**panic 带源文件路径 + 行号**
- 编译器能静态证明不为 null 时自动消除检查（零开销）
- 格式：`panic: null assertion failed at src/main.ch:42`

### 7. 后缀 `?` 传播运算符完整语义
- `T?` → null 时 `return null`，要求当前函数返回 `U?`
- `Result<T,E>` → Err 时 `return Err(e)`，要求当前函数返回 `Result<U,E>`
- 其他类型使用 `?` 是**编译错误**（包括 void/int 返回函数）
- `?` 和 `!` 在运算符优先级 Level 1（后缀，最高优先级组）
- 文档：`可空类型设计.md §2.3/2.5`、`错误处理设计.md §2.3`、`运算符优先级.md` 全部更新
